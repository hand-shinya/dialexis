# IMPROVEMENT_PROTOCOL.md — 改良手順書（systemによるsystemの改良）

対象読者：Dialexisに変更を加えようとするすべての人間とAI。本書は「不具合や改良点に対して、このsystem自身によってsystemを改良する」ための考え方と手順を定める。

## 0. 原理：改良自体を一つの研究として扱う（dogfooding）

Dialexisの中核主張は「問いは、主張・根拠・反証・未確認事項・判断のグラフとして扱うとより良く進む」である。**この主張を、自分自身の改良に適用できないなら、その主張は疑わしい。** よって：

> すべての非自明な改良提案は、Dialexisの研究デスク上に1つの研究プロジェクトとして立て、
> 「問い（question）→ 現状の主張（claim）→ 破綻の根拠（evidence）→ 反対説（counterclaim）
> → 未確認事項（uncertainty）→ 判断（decision: adopted/held/rejected）」として記録する。

これにより (1) 改良の来歴が残る、(2) 棄却された案が再提案されたとき理由ごと参照できる、(3) 本システムのUXの欠陥を開発者自身が最速で体感する。

## 1. スパイラル型高次元化弁証法（改良の思考手順）

局所パッチの前に、必ず次を1周する：

1. **破綻境界の特定** — いまの設計は「どの利用者が・どの状況で」使うと壊れるか。バグ報告を一般化し、境界条件として言い直す。
2. **問いの再定式化** — 「このバグを直すには？」ではなく「この破綻が示す、設計が置いた誤った前提は何か？」と問い直す。
3. **保存/棄却/上位化の区分** — 現設計から保存すべきもの・棄却すべきもの・より上位の概念へ引き上げるべきものを明示的に分ける。
4. **公理適合テスト** — 解決案を七公理（GENESIS第一部）に照らす。特に：
   - Level 0 が保たれるか（公理5）
   - 取得時刻の刻印が消えないか（公理4）
   - 内部への知識蓄積を始めていないか（公理3）
   - 利用者に新たな常時負担を課さないか（公理1・2）
5. **記録** — 判断とその理由を研究プロジェクトに、設計原理の変更ならGENESIS.md改版履歴に残す。

例（本リポジトリの起源そのものが一周目である）：「哲学データベースを作る」という初期構想は、鮮度と著作権の境界で破綻する → 問いを「何を保存すべきか」へ再定式化 → データは棄却・来歴は保存・「レンズ」へ上位化（GENESIS改版履歴 v1.0）。

## 2. 変更の分類と必要手続

| 分類 | 例 | 必要手続 |
|---|---|---|
| A. 修繕 | バグ修正、誤訳、表示崩れ | PR＋テスト通過のみ |
| B. 機能追加（公理内） | 新コネクタ、新UI、言語追加 | 研究プロジェクト1件＋公理適合テスト＋PR |
| C. 設計変更 | スキーマ変更、固定語彙の変更、依存追加 | B＋ARCHITECTURE.md更新＋移行手順 |
| D. 憲法変更 | 公理の追加・修正、目的の再定義 | C＋GENESIS.md第五部の改版手続（弁証法1周の記録必須） |

## 3. 実装規律

- **スキーマは追記的に変更する**（列追加・テーブル追加は可、破壊的変更は移行スクリプト必須）。理由：コード巻き戻しでデータが壊れない状態を保つ（OPERATIONS.md §4）。
- **依存の追加は正当化を要する**。「ビルド工程なし・フレームワークなし」は公理7の実装であり、便利さでは覆らない。
- **テスト**：新機能はオフラインの Level 0 経路を必ず `tests/` に追加する。ネットワーク依存テストは分離する。
- **AIによる改良**：AIが生成したコード・文書はPR説明に明記する（公理6はコードにも適用される）。AIは GENESIS.md → ARCHITECTURE.md → 本書の順で読み込んでから着手すること。

## 4. 問題の予測（既知の将来課題）

ROADMAP.md の各段階に対応する予測課題を先に登録しておく：

| 予測される問題 | 兆候 | 準備済みの対応方向 |
|---|---|---|
| 無料APIのレート制限・仕様変更 | explore のerrorバッジ常態化 | 統一封筒のおかげでコネクタ単位で修理可能。TTL延長・ミラー切替 |
| Watcher の検索ノイズ（例：query型でKarl Kraus混入） | 新着の適合率低下 | author型（OpenAlex ID紐付け）を優先UIに。フェーズ2で概念フィルタ |
| 公開研究痕跡のスパム・低品質流入 | フェーズ2開始後 | 公開はデフォルトOFF、署名つき公開、レビューの接ぎ木（公理2の順序） |
| SQLiteの限界 | 公開痕跡の横断検索が遅延 | JSON-LDエクスポート経由でGraph DBへ段階移行（ARCHITECTURE.md §3） |
| コミュニティLLMプールの濫用 | 実装した場合 | DONATIONS.md の設計（上限・認証・用途限定）以外で実装しない |
| 西洋中心・英語中心バイアス | 非西洋概念の探索結果が貧弱 | Wikidata多言語アンカーの活用、非西洋情報源コネクタの追加を優先度高で扱う |

## 5. リリース手順

1. `pytest` 全通過 → 2. ライブ検証3点（BUILD_FROM_ZERO.md §2）→ 3. git tag `vX.Y.Z` → 4. VPSで OPERATIONS.md §4 → 5. 変更点が利用者可視なら About/README 更新。
